# エラーハンドリング設計

> TagWatchman が失敗をどう扱うかの設計方針。
> 「なぜこのコードがこう振る舞うのか」を理解するためのガイドです。

このドキュメントは [Apache License 2.0](../LICENSE) の下で提供されます。

---

## 全体を貫く2原則

1. **握りつぶすのは想定内の不在だけ。それ以外の失敗は必ずどこかで可視化する。**
2. **例外を投げるか握るかは、それが主処理の成否を左右すべきかで決める。**

同じ「通知の送信失敗」が場所によって投げる/握るに分かれるのは、この2原則の現れです。
このシステムは利用者の AWS リソースを不可逆に操作しうるため、失敗を隠さないことを最優先に設計しています。

---

## 5つの層

エラー処理は役割ごとに5つの層に分かれます。

| 層 | 役割 | 方針 | スタンス |
|---|---|---|---|
| Lambda 内部 | boto3 例外の一次処理 | 不在(NotFound)は飲む、それ以外は整えて投げる | 選別 |
| Step Functions（Isolate） | 投げられた失敗を裁く | 操作が通るまでリトライ、`States.ALL` で全部拾って可視化 | 選別 |
| notifier | 検知/承認の通知 | 通知失敗は **投げる**（通知が主処理だから） | 投げる |
| deleter | 承認削除の結果通知 | 報告失敗は **握る**（結果メールは事後報告だから） | 握る |
| 自己保護ガード | 自スタックの保護 | 有害な操作は止めて通知・無害なものは静かに終える。実害の大きい操作ほど手前で止める | 選別 |

「選別」の3層が投げる/握るを振り分け、notifier と deleter はその結果として扱いが一方に定まります。
notifier(投げる)と deleter(握る)の対比がこの設計の要点で、通知の位置づけ(主処理か事後報告か)で扱いが逆になります。

---

## 各層の考え方

**Lambda 内部** — 握るのは「もう存在しないリソース」だけ。状態エラーや権限エラーは抱え込まず、上位の Step Functions に判断を委ねます。例外を投げるときは、上位層が正しく扱える固有の型に整えてから投げます。言い換えると、自分で抱え込まずに「これは想定内」「これは上位に判断させる」を仕分けるだけの層です。

**Step Functions（Isolate）** — リソースの状態確認(describe)は実態と一致しないことがあるため、状態を信じて待つのではなく「操作が実際に通るか」でリトライします。リトライで吸収できなかった失敗、リトライ対象外の失敗、想定外の例外は、すべて `States.ALL` の Catch が拾い、検知通知に変換します。無言で終了する経路を作らないのが狙いです。言い換えると、状態確認を当てにせず「実際にやってみて通るまで一定間隔で繰り返し、ダメだったものは漏れなく通知に回す」層です。

**notifier** — 利用者が事象を知る唯一の経路です。自分自身が壊れないこと(防御的な値の読み取り、件名のサニタイズ)と、壊れた事実を隠さないこと(送信失敗は例外を投げて実行ごと失敗させる)を両立させます。通知は主処理なので、届かなければ失敗として扱います。言い換えると、通知できなかったときに黙って終わらず、処理ごと失敗させて気づけるようにする層です。

**deleter** — 承認 URL のクリックを起点に、非同期で呼ばれます。削除に失敗しても例外で握りつぶさず、失敗ステータスを返したうえで失敗を通知し、再試行は承認 URL の再クリックに集約します。結果メールの送信はベストエフォートで、送信に失敗しても削除そのものの成否(ステータス)は変えません。言い換えると、削除という本処理はもう終わっているので、結果報告が届かなくても削除の成否そのものは書き換えない層です。

**自己保護ガード** — TagWatchman 自身のリソースを誤って隔離・削除しないためのガードです。実際の操作に入る前に弾き、隔離と削除の両方の入口に置くことで、削除パスまで確実に保護します。言い換えると、自分自身を壊しに行く操作を手前で止め、止めただけのときは静かに終え、本来届くはずのない危険な操作が来たときだけ異常として知らせる層です。

## 投げる / 握る 早見表

| 状況 | 扱い | 理由 |
|---|---|---|
| 対象が既に存在しない（NotFound） | 握る | 冪等。想定内の不在 |
| 状態エラー・権限エラー（隔離処理） | 投げる | 上位層に判断を委ねる |
| 通知の送信失敗（notifier） | 投げる | 通知は主処理 |
| 結果メールの送信失敗（deleter） | 握る | 事後報告。主処理は完了済み |
| 削除失敗（deleter） | 握る → 失敗ステータスを返す | 失敗を通知し、再試行は承認 URL に集約 |
| 自スタックのリソース（自己保護） | 握る（正常終了） | 保護は正常系。異常通知はノイズになる |

---

## 設計を支える判断基準

- **冪等性を保つ**: 同じ処理が再実行されても結果が壊れないようにする。リトライを導入する箇所には、再入されても元の状態を破壊しないガードをセットで用意します。

- **挙動を消す前に、それが何を支えているか問う**: 一見不要に見える挙動が、別の安全性を担保していることがあります。たとえば承認後も実行が一定期間継続するのは、削除失敗時の再試行経路として機能する意図的な仕様です。
