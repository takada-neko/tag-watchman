# リソース検知後の挙動一覧
 
TagWatchmanがタグ不足リソースを検知した後の挙動をリソース別にまとめています。
 
## 挙動パターン
 
| パターン | 内容 |
|---|---|
| **A. 隔離→承認→削除** | 即時隔離（操作遮断） → 7日間の猶予 → 承認メール → 人間が承認 → 削除 |
| **B. 条件付き即時削除** | 条件に応じて即時削除または通知のみ |
| **C. 権限剥奪→承認→削除** | 即時権限剥奪 → 7日間の猶予 → 承認メール → 人間が承認 → 削除 |
| **D. 専用フロー** | タグ関係なく即時対応（CloudTrail専用） |
| **E. 通知のみ** | 検知・通知のみ。隔離・削除は行わない |
 
> **共通ルール：タグが付いているリソースは一切対象になりません。**
> TagWatchmanは「タグが不足している」ことのみをトリガーに動作します。
 
---
 
## 隔離リソースへのタグ付与（復旧のトリガー）
 
パターンA・Cで隔離されたリソースは、**必須タグを付与すると自動的に復旧**します。ただし、誰でもタグを付与できると、攻撃者が自分でタグを付けて隔離を解除できてしまいます。
 
そこでTagWatchmanは、隔離リソースへのタグ付与を**専用のオペレーターロール（`tagwatchman-operator`）経由でのみ許可**します。
 
```
通常のユーザー・管理者            → タグ付与は拒否される
tagwatchman-operator にスイッチロール → タグ付与が可能（＝復旧のトリガー）
```
 
`tagwatchman-operator` にスイッチロールできるIAMエンティティは、デプロイ時の `TrustedPrincipals` パラメータで指定します（複数指定可・任意でIP制限可）。
 
> **補足：タグ付与制限が厳密に効くのはリソースポリシー対応サービス（S3 / DynamoDB / SQS / SNS / Kinesis / ECR / Step Functions）です。**
> SGの差し替えで隔離するサービス（EC2 / RDS など）のタグ付与はIAMで制御されるため、該当リソースのタグ付与権限を持つIAMエンティティであれば付与可能です。運用上はタグ付与権限を絞ることを推奨します。

## パターンA：隔離→承認→削除
 
主要操作を遮断して無害化した上で、7日間の猶予期間を設けます。猶予期間中に（オペレーターロール経由で）タグを付与すれば自動で復旧します。
 
```
検知 → 即時隔離（操作遮断）→ メール① → 7日間の猶予
    ↓ タグ付与（operator経由）→ 自動復旧（隔離解除）
    ↓ タグなし → メール②（承認依頼）→ 人間が承認 → 削除
```
 
各サービスとも、隔離は「データの読み書きや主要な操作を拒否する」もので、タグ付与など復旧に必要な操作は許可されています。これにより「害を与えられない状態」にしつつ、復旧の導線を残しています。
 
| サービス | 隔離方法 | 復旧方法 |
|---|---|---|
| EC2 | SGを全拒否SGに差し替え | タグ付与で元のSGに戻す |
| RDS | SGを全拒否SGに差し替え | タグ付与で元のSGに戻す |
| S3 | バケットポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| Lambda | 同時実行数を0に設定 | タグ付与で同時実行数を復元 |
| DynamoDB | リソースポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| SQS | キューポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| ECS | SGを全拒否SGに差し替え | タグ付与で元のSGに戻す |
| EKS | クラスターのSGを全拒否SGに差し替え | タグ付与で元のSGに戻す |
| SNS | トピックポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| Kinesis | リソースポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| OpenSearch | アクセスポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| ECR | リポジトリポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| Redshift | SGを全拒否SGに差し替え | タグ付与で元のSGに戻す |
| Step Functions | リソースポリシーで主要操作を拒否 | タグ付与でポリシーを削除 |
| API Gateway | ステージを削除してエンドポイント無効化 | タグ付与でステージを再作成 |
 
> **隔離中の復旧操作について：**
> 復旧を行うLambda（restorer）は、隔離ポリシーの対象から除外されています。これによりタグ付与をトリガーに、ポリシー削除などの復旧操作を実行できます。

## パターンB：条件付き即時削除
 
ネットワーク系リソースは「存在するだけでリスク」になるため、条件に応じて即時削除します。
 
### IGW（Internet Gateway）
 
```
検知
    ↓
VPCにアタッチされているか確認
    ├─ アタッチなし → 即時削除
    └─ アタッチあり → 通知のみ（手動対応が必要）
```
 
> **アタッチありの場合に自動削除しない理由：**
> 既存VPCへのアタッチを解除すると、そのVPC内の全リソースがインターネット通信できなくなります。本番環境への影響が大きすぎるため、人間の判断に委ねています。
 
### NAT Gateway
 
```
検知 → 即時削除
```
 
タグなしのNAT Gatewayは管理外リソースとみなして即時削除します。
 
### VPC Peering
 
```
検知 → 即時削除
```
 
タグなしのVPC Peeringは不正な横断アクセスの経路になりうるため即時削除します。
 
### EIP（Elastic IP）
 
```
検知
    ↓
インスタンス・ネットワークインターフェイスにアタッチされているか確認
    ├─ アタッチなし → 即時解放（Release）
    └─ アタッチあり → 通知のみ（手動対応が必要）
```
 
> **アタッチありの場合に自動解放しない理由：**
> アタッチされているEIPを解放すると、関連するEC2インスタンスなどの固定IPが失われます。本番環境への影響が大きいため、人間の判断に委ねています。

## パターンC：権限剥奪→承認→削除
 
IAMリソースは隔離の代わりに「権限の剥奪」を行います。削除後の復旧が難しいため、**人間の承認なしに削除は行いません。**
 
```
検知 → 即時権限剥奪 → メール①（手動対応の案内付き）→ 7日間の猶予
    ↓ タグ付与 → 隔離タグを削除（ポリシーの再付与は手動）
    ↓ タグなし → メール②（承認依頼）→ 人間が承認 → 削除
```
 
### IAM Role
 
| 自動対応 | 内容 |
|---|---|
| マネージドポリシーを全剥奪 | アタッチされている全ポリシーをデタッチ |
| インラインポリシーを全削除 | ロールに直接記述されたポリシーを削除 |
 
> **⚠️ 復旧は手動対応が必要です**
> タグを付与しても剥奪されたポリシーは自動で戻りません。メール通知に手順が記載されます。
 
### IAM User
 
| 自動対応 | 内容 |
|---|---|
| マネージドポリシーを全剥奪 | アタッチされている全ポリシーをデタッチ |
| インラインポリシーを全削除 | ユーザーに直接記述されたポリシーを削除 |
| アクセスキーを全て無効化 | 発行済みの全アクセスキーをInactiveに設定 |
 
> **⚠️ 復旧は手動対応が必要です**
> タグを付与してもポリシーの再付与・アクセスキーの再作成は手動で行う必要があります。メール通知に手順が記載されます。

## パターンD：専用フロー（CloudTrail Guardian）
 
CloudTrailはTagWatchman自体の生命線です。タグの有無に関わらず、操作を検知したら即時対応します。
 
| イベント | 重大度 | 自動対応 |
|---|---|---|
| StopLogging | 🚨🚨🚨 CRITICAL | 自動で再有効化 + 最強警告メール |
| DeleteTrail | 🚨🚨🚨 CRITICAL | 最強警告メール（再作成は手動） |
| UpdateTrail | ⚠️ WARNING | 警告メール |
| PutEventSelectors | ⚠️ WARNING | 警告メール |
 
> **StopLoggingだけ自動再有効化できる理由：**
> Trailは削除されていないため `StartLogging` を呼ぶだけで復旧できます。DeleteTrailはTrail自体がなくなるため、S3バケット名などの設定情報が不明になり自動再作成ができません。

## パターンE：通知のみ
 
影響範囲が大きすぎる、あるいはサービスの特性上ネットワーク隔離が適さないため、自動隔離・削除は行わず通知のみです。
 
| サービス | 通知のみの理由 |
|---|---|
| VPC | 削除すると内部の全リソースの通信が止まる。影響範囲が大きすぎる |
| ElastiCache     | サービスの特性上、確実な隔離が保証できないため通知のみ |
| Glue            | リソースポリシーがデータカタログ全体に適用され、database単位の隔離ができない |
| Workspaces      | サービスの特性上、ネットワーク隔離が適さないため通知のみ |
| Glue | リソースポリシーがデータカタログ全体に適用されるため、database単位での隔離が不可能 |
| IGW（アタッチあり） | 既存VPCの通信が止まる可能性がある |
| EIP（アタッチあり） | 関連リソースの固定IPが失われる |

## よくある質問
 
**Q. 誤検知でリソースが隔離された場合はどうすればいいですか？**
A. 猶予期間（デフォルト7日）中に、`tagwatchman-operator` ロールにスイッチした上で必須タグを付与してください。タグが付与された時点で自動的に隔離が解除されます。
 
**Q. なぜ普通の管理者権限ではタグを付与できないのですか？**
A. 誰でもタグを付与できると、攻撃者が自分でタグを付けて隔離を解除できてしまうためです。タグ付与は `tagwatchman-operator` ロール経由に限定し、このロールを使える人を `TrustedPrincipals` で絞ることで、復旧操作を信頼できる人だけに制限しています。
 
**Q. IAMロールのポリシーが剥奪されてしまいました。どう復旧しますか？**
A. タグを付与した後、マネジメントコンソールまたはCLIで必要なポリシーを手動で再アタッチしてください。TagWatchmanからのメール①に対応手順が記載されています。
 
**Q. 猶予期間を7日より短く・長くしたいです。**
A. SSM Parameter Storeの `/tagwatchman/delete-delay-seconds` を変更してください。再デプロイ不要で即時反映されます。
 
**Q. 特定のリソースだけTagWatchmanの対象外にしたいです。**
A. 必須タグを付与することで対象外になります。TagWatchmanは「タグが不足している」ことのみをトリガーにしています。
