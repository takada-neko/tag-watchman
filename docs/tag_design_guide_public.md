# AWSタグ設計ガイド（基本編）

## はじめに

このガイドはAWSリソースのタグ設計を体系化するための実践的なドキュメントです。個人・スタートアップから中小企業まで、タグ設計のベストプラクティスをまとめています。

TagWatchmanと組み合わせることで、タグ設計→自動検知→隔離→削除という一連のガバナンスサイクルを実現できます。

> **📦 Terraform実装例・サービス別注意点・SSM連携設定などを含む詳細版は有料セットに含まれています。**

## ① なぜタグ設計が重要か

### コスト管理

タグがないと「誰が何のために使っているか」がわからないまま請求が積み上がります。

```
# タグありの場合
Project=my-api, Env=prod → APIサーバーの本番環境コスト
Project=my-api, Env=stg  → APIサーバーのステージング環境コスト

# タグなしの場合
EC2: $XXX.XX ← 何のインスタンス？誰が作った？
```

AWSのコスト配分タグを設定すれば、プロジェクト別・環境別のコストをCSVで出力できます。

### セキュリティ・ガバナンス

**「存在を把握できていないリソース」は最大のリスクです。**

- 離任したメンバーが作ったリソースが放置される
- 攻撃者がバックドアとして作ったインスタンスを見逃す
- パッチが当たらない・監査されない・ログが見られない

タグ設計を徹底することで、全リソースの「誰が・何のために・いつ作ったか」を把握できます。

### TagWatchmanとの連携

TagWatchmanはタグが不足しているリソースを自動検知・隔離します。つまり**タグ設計 = セキュリティポリシー**として機能します。

```
タグ設計書を作成
    ↓
TagWatchmanのSSMパラメータに反映
    ↓
タグなしリソースを自動検知・隔離
    ↓
組織全体でタグポリシーが強制される
```

## ② タグ設計の基本原則

### 命名規則

**キーの形式**

```
# 推奨：小文字・ハイフン区切り
env, project, owned-by

# または：PascalCase
Env, Project, OwnedBy

# 統一することが最重要（混在はNG）
env と Env を混在させない
```

**値の形式**

```
# 推奨：小文字・ハイフン区切り
prod, stg, test
my-project-api
backend-team
```

### 必須タグ vs オプションタグ

| 種類 | 説明 | 例 |
|---|---|---|
| **必須タグ** | 全リソースに必ず付ける | Env, Project, Owned |
| **推奨タグ** | できれば付ける | CostCenter, Terraform |
| **任意タグ** | 必要に応じて付ける | Version, ExpiresAt |

**必須タグは最小限に絞ることが重要です。多すぎると付け忘れが増えます。**

### 値の標準化

```
# 良い例：選択式（許可値を定義する）
Env: prod | stg | test

# 悪い例：自由入力（表記ゆれが発生する）
Env: production / Production / PROD / prod
```

TagWatchmanのSSMパラメータで許可値を定義することで、表記ゆれを防げます。

## ③ 個人・スタートアップ向け（3タグ構成）

**シンプルに始めることが最重要です。**

| タグキー | 説明 | 許可値 | マッチ方式 |
|---|---|---|---|
| `Env` | 環境 | prod, stg, test | 完全一致 |
| `Project` | プロジェクト名 | your-project | 前方一致 |
| `Owned` | 担当チーム・担当者 | 空文字以外ならOK | 空文字チェックのみ |

**タグ付け例：**

```
Env     = prod
Project = my-api
Owned   = backend
```

> 詳細な設定方法（許可値の設定・判定ルールなど）は購入者向けのドキュメントに記載しています。
